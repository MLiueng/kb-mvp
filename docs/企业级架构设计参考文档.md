## 一、高可用架构设计

### 1.1 高可用目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| SLA | 99.95% | 年停机 ≤ 4.38 小时 |
| RTO（恢复时间目标） | ≤ 15 分钟 | 故障到恢复的最大时长 |
| RPO（恢复点目标） | ≤ 1 分钟 | 最大数据丢失窗口 |
| 单点故障消除 | 全链路无单点 | 每层至少 2 副本 |

**故障域划分原则**：从可用区（AZ）→ K8s 节点 → Pod 三个层次隔离故障域，核心服务跨 AZ 部署，Pod 反亲和调度避免同节点集中。

### 1.3 微服务高可用

#### 无状态服务多副本 + 反亲和调度

```yaml
# K8s 部署配置：Pod 反亲和，确保同服务 Pod 分布在不同节点
apiVersion: apps/v1
kind: Deployment
metadata:
  name: kb-service
spec:
  replicas: 3                      # 至少 3 副本
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0            # 滚动更新时保证可用副本不减少
  template:
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchLabels:
                  app: kb-service
              topologyKey: kubernetes.io/hostname   # 不同节点
      containers:
      - name: kb-service
        image: kb-service:v1.0
        readinessProbe:            # 就绪探针确保流量只打到健康实例
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
        livenessProbe:             # 存活探针自动重启不健康实例
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 60
          periodSeconds: 20
```

#### 服务熔断降级（Sentinel 规则设计）

| 规则类型 | 场景 | 配置 |
|----------|------|------|
| 慢调用熔断 | RAG 问答 LLM 调用超时 | RT > 10s，比例 > 50%，熔断 30s |
| 异常比例熔断 | 向量检索异常 | 异常比例 > 50%，熔断 10s |
| 流控规则 | 文档上传并发限制 | QPS 100，排队等待 5s |
| 降级规则 | ES 不可用时降级 | 异常数 > 10，降级返回缓存结果 |

#### 优雅上下线与流量无损发布

1. **下线**：preStop Hook → 从注册中心注销 → 等待 30s 处理存量请求 → 退出
2. **上线**：启动 → 就绪探针通过 → 注册中心注册 → 流量进入

### 1.4 数据层高可用

| 组件 | 高可用方案 | 故障切换机制 |
|------|-----------|-------------|
| MySQL | 主从复制 + MHA/Orchestrator | 主库故障自动提升从库，30s 内切换 |
| Redis | Cluster 模式（3主3从） | 哨兵监控，主节点故障自动故障转移 |
| Elasticsearch | 集群分片 + 副本（replicas≥1） | 节点故障自动重平衡，副本提升为主 |
| Milvus | 集群多节点 + QueryNode/QueryCoord | 节点故障自动剔除，副本接管查询 |
| MinIO | 纠删码模式（4节点+） | 节点故障数据可恢复，自动修复 |
| RocketMQ | Dledger 集群（3节点） | 主节点故障自动选主 |

```
                    ┌──────────────────┐
                    │  推理负载均衡器    │
                    │  (Nginx/Triton)  │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ vLLM #1  │  │ vLLM #2  │  │ vLLM #3  │
```

- **多实例负载均衡**：vLLM 部署多实例，通过 Nginx/Triton Inference Server 轮询分发
- **健康检查**：定期推理测试请求，异常实例自动摘除
- **模型热加载**：新模型版本加载至备用实例，验证后切流，旧实例回收
- **故障转移**：实例宕机由负载均衡器自动剔除，剩余实例承接流量

## 二、数据容灾备份方案

### 2.1 备份策略总览

| 数据类型 | 备份方式 | 频率 | 保留周期 | 存储位置 |
|----------|----------|------|----------|----------|
| MySQL | xtrabackup 全量 + binlog 增量 | 全量日1次/增量实时 | 30天 | 本地 + 异地 |
| MinIO 文档 | 跨区复制 + 快照 | 实时复制 | 90天版本 | 异地 MinIO |
| Milvus 向量 | snapshot + 原文重建 | 周全量 | 30天 | 本地 + 异地 |
| ES 索引 | snapshot API | 日增量 | 14天 | 本地 |
| Redis | RDB 快照 + AOF | RDB 6h/AOF 实时 | 7天 | 本地 |
| Nacos 配置 | 导出 YAML | 变更触发 | 90天 | Git 仓库 |
| K8s Secret | Vault 备份 | 日全量 | 90天 | 加密异地 |

**备份原则**：3-2-1 原则（3 份数据、2 种介质、1 份异地）。

### 2.2 数据库备份

#### MySQL 物理备份 + binlog 增量

```bash
# 全量备份（每日凌晨）
xtrabackup --backup --target-dir=/backup/mysql/full_$(date +%Y%m%d) \
  --user=root --password=$MYSQL_PWD

# 增量备份（每小时，基于 binlog）
mysqlbinlog --read-from-remote-server --raw --stop-never \
  --host=$MYSQL_HOST --user=repl --password=$REPL_PWD \
  --result-dir=/backup/mysql/binlog/ mysql-bin.000001

# 异地传输
rsync -avz --delete /backup/mysql/ backup@dr-site:/dr/mysql/
```

#### 恢复演练流程

1. 在隔离环境恢复全量备份
2. 回放 binlog 至指定时间点（PITR）
3. 数据一致性校验（行数/校验和比对）
4. 应用层功能验证
5. 记录恢复耗时，验证 RTO/RPO 达标

### 2.3 对象存储备份

**MinIO 跨区复制**：
```bash
# 配置跨区复制规则
mc replicate add source/doc-bucket dr-site/doc-bucket \
  --remote-bucket "dr-bucket" \
  --replicate "delete,delete-marker,existing-objects"
```

**文档版本保留**：MinIO 对象版本控制（Object Versioning），保留历史版本 90 天，支持误删恢复。

### 2.4 向量库备份

**Milvus snapshot**：
```python
# 创建集合快照
utility.create_snapshot(collection_name="doc_chunks", snapshot_name="weekly_$(date)")
# 恢复快照
utility.restore_snapshot(snapshot_name="weekly_20260101")
```

**Embedding 重建策略**：向量数据可从原始文档重新生成（保留原文 + Embedding 模型即可重建），作为终极容灾手段。MVP 验证版的多供应商适配层支持切换 Embedding 模型后重建。

### 2.5 配置与密钥备份

- **Nacos 配置**：定期导出为 YAML，提交至 Git 仓库（配置中心版本管理）
- **K8s Secret / Vault**：Vault 快照备份至加密异地存储；K8s Secret 通过 Velero 整体备份

### 2.6 灾难恢复演练

| 项目 | 内容 | 频率 |
|------|------|------|
| 数据库恢复 | 全量+binlog 恢复至隔离环境 | 每月 |
| 对象存储恢复 | 跨区复制验证 + 误删恢复 | 每季度 |
| 向量库恢复 | snapshot 恢复 + 重建验证 | 每季度 |
| 全链路灾备 | 模拟 AZ 故障，切换至灾备站点 | 每半年 |
| RTO/RPO 验证 | 计时恢复全过程，验证达标 | 每次演练 |

**恢复 SOP**：故障通告 → 评估范围 → 启动恢复 → 数据校验 → 服务恢复 → 事后复盘。

## 三、权限与角色管控体系

### 3.1 权限模型总体设计

```
┌─────────────────────────────────────────────────────────────┐
│                  RBAC + ABAC + ACL 三层混合模型               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第一层：RBAC（角色权限）                                     │
│  ├── 用户 → 角色 → 功能权限                                   │
│  ├── 控制能做什么操作（如：文档上传、知识库创建）               │
│  └── 粗粒度，基于角色                                         │
│                                                             │
│  第二层：ABAC（属性权限）                                     │
│  ├── 基于用户/资源/环境属性动态决策                            │
│  ├── 控制在什么条件下能做（如：仅工作时间、仅本部门）           │
│  └── 细粒度，基于策略                                         │
│                                                             │
│  第三层：ACL（访问控制列表）                                   │
│  ├── 文档级显式权限                                           │
│  ├── 控制能访问哪个具体文档                                    │
│  └── 最细粒度，基于资源                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**各层职责**：RBAC 管功能权限（能做什么），ABAC 管数据范围（能看哪些），ACL 管具体资源（能访问哪个文档）。

### 3.3 组织架构与数据权限

#### 组织树设计

```
集团（Root）
├── 子公司 A
│   ├── 技术部
│   │   ├── 研发一组
│   │   └── 研发二组
│   ├── 产品部
│   └── 市场部
├── 子公司 B
│   ├── 技术部
│   └── 财务部
```

#### 数据权限范围

| 范围 | 说明 | SQL 过滤条件 |
|------|------|-------------|
| 本人 | 仅看自己创建的数据 | `creator = #{userId}` |
| 本部门 | 本部门所有数据 | `dept_id = #{deptId}` |
| 本部门及子部门 | 含下级部门 | `dept_id IN (SELECT id FROM dept WHERE path LIKE '#{deptPath}%')` |
| 全部 | 本租户全部数据 | `tenant_id = #{tenantId}` |

**数据权限 SQL 拦截**：基于 MyBatis-Plus 数据权限拦截器，自动在 SQL 追加范围条件，开发者无需手动处理。

### 3.4 文档级权限管控

#### ACL 权限矩阵

| 权限 | 编码 | 说明 |
|------|------|------|
| 阅读 | `READ` | 查看文档内容 |
| 编辑 | `WRITE` | 修改文档内容 |
| 删除 | `DELETE` | 删除文档 |
| 分享 | `SHARE` | 分享文档给他人 |
| 下载 | `DOWNLOAD` | 下载文档原文 |
| 打印 | `PRINT` | 打印文档 |

#### 权限继承与覆盖规则

1. 知识库权限 → 文档权限（继承）：文档默认继承所属知识库的权限
2. 文档级 ACL 覆盖：对特定文档设置 ACL 可覆盖继承权限
3. 显式拒绝优先：`DENY` 权限优先于 `ALLOW`（黑名单优先）
4. 敏感文档标记：标记为"机密"的文档强制需要显式授权，不继承

#### 敏感文档审批流

```
标记敏感文档 → 提交审批 → 部门负责人审核 → 安全管理员终审 → 授权生效
```

### 3.5 权限校验流程

```
请求到达
  │
  ▼
API Gateway：Token 解析与验证（JWT 签名/过期检查）
  │
  ▼
Auth 服务：加载用户角色与权限（Redis 缓存）
  │
  ▼
RBAC 校验：用户角色是否拥有该 API 的功能权限？
  │ ├─ 否 → 403 Forbidden
  │ └─ 是
  ▼
ABAC 策略评估：属性条件是否满足？（时间/IP/部门）
  │ ├─ 否 → 403 Forbidden
  │ └─ 是
  ▼
数据权限过滤：SQL 拦截器追加数据范围条件
  │
  ▼
ACL 检查（文档级）：目标文档是否允许访问？
  │ ├─ 否 → 404 Not Found（不暴露存在性）
  │ └─ 是
  ▼
请求处理 → 响应
  │
  ▼
审计日志记录（异步）
```

**权限缓存策略**：用户角色与权限加载后缓存至 Redis（TTL 30 分钟），权限变更时主动失效缓存。

### 3.6 权限审计与合规

- **权限变更审计**：所有角色分配/ACL 修改记录审计日志（操作人/时间/变更内容）
- **定期权限审查**：每季度生成权限报告，租户管理员复核
- **越权访问告警**：连续 403 响应 > 5 次/分钟触发安全告警
- **最小权限原则**：默认无权限，显式授权

### 3.7 SSO 与身份联邦

| 协议 | 场景 | 对接方式 |
|------|------|----------|
| LDAP/AD | 企业内部域账号 | Spring Security LDAP 认证 |

### 4.3 日志体系

### 4.4 告警规则设计

#### 告警分级

| 级别 | 定义 | 响应时间 | 通知方式 |
|------|------|----------|----------|
| P0 紧急 | 核心服务不可用 | 5 分钟 | 电话 + 短信 + 钉钉 |
| P1 重要 | 核心功能降级 | 15 分钟 | 短信 + 钉钉 |
| P2 一般 | 非核心功能异常 | 1 小时 | 钉钉 + 邮件 |
| P3 提示 | 容量预警 | 4 小时 | 邮件 |

#### 核心告警规则

| 指标 | 阈值 | 持续时间 | 级别 |
|------|------|----------|------|
| 服务可用率 | < 99% | 2 分钟 | P0 |
| API P99 延迟 | > 5s | 3 分钟 | P1 |
| API 错误率 | > 5% | 2 分钟 | P1 |
| MySQL 主从延迟 | > 10s | 1 分钟 | P1 |
| Redis 内存使用 | > 85% | 5 分钟 | P2 |
| ES 集群状态 | red | 即时 | P0 |
| Milvus 检索延迟 | > 500ms | 3 分钟 | P2 |
| GPU 显存使用 | > 95% | 5 分钟 | P2 |
| 磁盘使用率 | > 80% | 10 分钟 | P2 |
| RocketMQ 消息堆积 | > 10000 | 5 分钟 | P1 |

#### 告警收敛与去重

- 同一告警 5 分钟内只通知一次（防风暴）
- 关联告警合并（如 MySQL 主从延迟 + 应用报错合并为一条）
- 维护窗口抑制（发布期间自动抑制相关告警）

### 4.5 告警处理流程

```
告警触发 → AlertManager 路由 → 通知值班人 → 值班人响应(ACK) → 排查处理 → 恢复确认 → 事后复盘
                │                                                    │
                └── 升级：15分钟未ACK → 升级至二级值班 ────────────────┘
```

**On-Call 轮值**：7×24 轮值排班，一级值班（开发）→ 二级值班（架构师）升级机制。

**故障复盘**：P0/P1 故障 48 小时内完成复盘报告（时间线/根因/改进措施/责任人）。

## 五、平滑升级策略

### 5.1 升级策略总览

| 策略 | 原理 | 适用场景 | 停机时间 |
|------|------|----------|----------|
| 滚动更新 | 逐个替换 Pod，新旧并存 | 常规版本升级 | 零 |
| 蓝绿部署 | 两套环境切换 | 大版本升级 | 零（切换瞬间） |
| 金丝雀发布 | 小流量验证后全量 | 高风险变更 | 零 |

### 5.2 数据库平滑迁移

#### DDL 变更兼容性原则

1. **扩展而非收缩**：先加列、后删列（分两次发布）
2. **避免锁表**：大表 DDL 使用 gh-ost/pt-osc 在线工具
3. **兼容期**：新旧代码兼容的中间 Schema 必须存在一个版本周期

#### 在线 DDL 工具

```bash
# gh-ost 在线表结构变更（不锁表）
gh-ost --host=mysql-master --user=root --password=$PWD \
  --database=knowledge_base --table=documents \
  --alter="ADD COLUMN category VARCHAR(64)" \
  --execute
```

#### 数据迁移双写方案

```
阶段1：双写（新表+旧表同步写入）→ 阶段2：数据回填（旧数据迁移新表）→ 阶段3：读切换（读新表）→ 阶段4：停止旧表写入
```

### 5.3 微服务滚动升级

```yaml
# K8s 滚动更新配置
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1           # 滚动时最多多出 1 个副本
    maxUnavailable: 0     # 滚动时不可用副本为 0（零停机）
```

**API 版本兼容**：v1/v2 接口并存，通过 Gateway 路由逐步切流，v1 观察期后下线。

**流量逐步切换**：金丝雀发布先导 5% 流量至新版本，观察指标无异常后逐步扩大至 100%。

### 5.4 向量库升级与数据迁移

- **Milvus 版本升级**：滚动升级 Milvus 集群组件，快照备份后升级
- **向量数据迁移**：导出 collection 数据 → 新版本导入 → 一致性校验（行数/抽样比对）
- **回滚**：保留旧版本快照，异常时秒级回滚

### 5.5 LLM 模型热切换

```
新模型加载至备用GPU实例 → A/B测试(5%流量) → 效果评估 → 灰度切换(20%→50%→100%) → 旧实例回收
```

**Embedding 模型切换的向量重建**：
1. 新 Embedding 模型部署
2. 全量文档重新向量化（异步任务，RocketMQ 驱动）
3. 双向量库并存（旧库继续服务，新库构建）
4. 新库构建完成 + 校验通过 → 切换检索至新库
5. 旧库下线

> MVP 验证版的向量一致性校验机制（记录 embedding 供应商/模型/维度）可直接复用于正式版，防止切换期间混用。

### 5.6 回滚机制

#### 回滚决策标准

| 条件 | 动作 |
|------|------|
| 新版本错误率 > 10% | 立即回滚 |
| 新版本 P99 延迟 > 旧版本 2 倍 | 立即回滚 |
| 核心功能不可用 | 立即回滚 |
| 金丝雀阶段异常 | 停止发布，回滚金丝雀 |

#### 回滚分类

| 类型 | 方式 | 耗时 |
|------|------|------|
| 应用回滚 | K8s `kubectl rollout undo` | < 1 分钟 |
| 数据库回滚 | DDL 反向操作 / binlog 回放 | 视数据量 |
| 向量库回滚 | 快照恢复 | < 5 分钟 |
| 配置回滚 | Nacos 版本回退 | < 1 分钟 |

**回滚 SOP**：告警/人工判断 → 决策回滚 → 执行回滚 → 验证恢复 → 通知相关人员 → 复盘。

## 六、总结与实施建议

### 6.1 五大模块优先级排序

| 优先级 | 模块 | 理由 |
|--------|------|------|
| P0 | 权限与角色管控 | 企业级安全基石，无权限不生产 |
| P0 | 监控告警体系 | 可观测性是运维前提，无监控不运维 |
| P1 | 高可用架构 | 生产稳定性保障 |
| P1 | 数据容灾备份 | 数据安全底线 |
| P2 | 平滑升级策略 | 持续演进保障，初期可简化 |

### 6.2 分阶段实施路线图

```
阶段一（第1-2周）：监控告警体系搭建
├── Prometheus + Grafana + Loki 部署
├── 基础设施 + 中间件 Exporter 接入
└── 核心告警规则与通知渠道配置

阶段二（第3-5周）：权限与角色管控
├── RBAC 角色体系与权限矩阵实现
├── 数据权限拦截器（MyBatis-Plus）
├── 文档级 ACL 与权限校验流程
└── SSO 集成（LDAP/OAuth2）

阶段三（第6-8周）：高可用架构
├── 多副本部署 + 反亲和调度
├── 数据层主从/集群高可用
├── Sentinel 熔断降级规则
└── GPU 推理多实例负载均衡

阶段四（第9-10周）：数据容灾备份
├── 数据库全量+增量备份
├── 对象存储跨区复制
├── 向量库快照机制
└── 首次灾备演练

阶段五（持续）：平滑升级策略
├── CI/CD 流水线完善
├── 金丝雀发布机制
├── 回滚自动化
└── 定期演练与优化
```

### 6.3 风险提示

1. **权限体系复杂度**：RBAC+ABAC+ACL 三层模型实现成本高，建议先实现 RBAC + 数据权限，ABAC/ACL 分阶段引入
2. **分布式事务**：跨服务数据一致性需 Seata 保障，注意性能开销与超时设计
3. **GPU 资源瓶颈**：vLLM 显存有限，需合理规划并发与模型选型，预留扩容空间
4. **Milvus 运维门槛**：分布式 Milvus 运维复杂，建议先单节点验证再逐步扩集群
5. **监控告警噪音**：初期告警规则需持续调优，避免告警疲劳

> **总结**：本文档从高可用架构、数据容灾备份、权限与角色管控、监控告警体系、平滑升级策略五个维度，为企业级私有知识库系统的落地实施提供了完整的设计指引。各模块可按优先级分阶段实施，与 MVP 验证版的技术结论（多供应商适配、向量一致性校验、分块策略）形成完整的演进链路。
