## 一、评估框架与方法论

### 1.1 四大评估维度

| 维度 | 定义 | 关键指标 |
|------|------|----------|
| **性能** | 系统在负载下的响应速度与吞吐能力 | QPS、延迟(P50/P99)、并发能力、资源利用率 |
| **扩展性** | 水平/垂直扩展能力与集群规模上限 | 分片机制、节点上限、自动扩缩容、数据重平衡 |
| **安全性** | 数据保护、访问控制、合规适配能力 | 加密支持、权限模型、审计能力、国密/等保适配 |
| **维护成本** | 运维复杂度、学习曲线、社区与人才储备 | 部署难度、监控成熟度、文档质量、人才可获得性 |

### 1.2 评分标准

采用 **5 星制**（★~★★★★★），按本项目（企业级私有知识库、多租户、RAG 问答）的实际需求加权：

| 权重等级 | 适用维度 | 权重 |
|----------|----------|------|
| 高 | 安全性、维护成本、扩展性 | ×1.2 |
| 中 | 性能 | ×1.0 |

### 1.3 评估原则

1. **场景驱动**：所有评分基于"企业级私有知识库 + RAG 问答"场景，非通用基准
2. **私有化优先**：优先评估私有化部署能力，云端托管方案仅作参照
3. **长期演进**：考虑 5+ 年生命周期内的可维护性与生态延续性
4. **客观中立**：基于事实数据与社区共识，避免主观偏好

## 二、后端框架选型对比

### 2.1 候选方案概览

| 方案 | 语言 | 定位 | 代表生态 |
|------|------|------|----------|
| **Spring Boot 3 + Spring Cloud Alibaba** | Java 17 | 企业级微服务首选 | Nacos/Sentinel/Seata/Dubbo |
| Quarkus | Java | 云原生超轻量框架 | Vert.x/Kotlin 生态 |
| FastAPI | Python | AI/快速原型 | Pydantic/Starlette |
| Gin + Kratos | Go | 高并发云原生 | gRPC/K8s 生态 |

### 2.2 多维度评分

| 维度 | Spring Boot 3 | Quarkus | FastAPI | Gin+Kratos |
|------|:---:|:---:|:---:|:---:|
| 性能 | ★★★★ | ★★★★★ | ★★★ | ★★★★★ |
| 扩展性 | ★★★★★ | ★★★★ | ★★ | ★★★★ |
| 安全性 | ★★★★★ | ★★★★ | ★★★ | ★★★★ |
| 维护成本 | ★★★★★ | ★★★ | ★★★ | ★★★ |
| **加权得分** | **4.84** | **3.96** | **2.88** | **3.84** |

### 2.3 详细分析

#### 性能

Spring Boot 3 基于 JVM JIT，长跑服务性能接近原生；Quarkus 采用编译时优化 + GraalVM Native Image，启动极快、内存占用低，但 Native Image 限制反射使用；FastAPI 基于 asyncio 异步 IO 性能良好，但 GIL 限制 CPU 并行；Go 原生编译，并发性能最优。

本项目性能瓶颈在 Milvus/ES/vLLM 等外部组件，后端承担业务编排，Spring Boot 性能完全满足 500 QPS。

#### 扩展性

Spring Cloud Alibaba 提供完整微服务治理（注册发现、配置中心、网关、熔断、分布式事务），水平扩展开箱即用。Quarkus 生态较新，治理组件不如 Spring Cloud 完善。FastAPI/Go 需大量自研微服务基础设施。

#### 安全性

Spring Security 是工业级安全框架，支持 OAuth2/JWT/RBAC/方法级权限，国密适配成熟（BC-GM）。Quarkus 安全体系基于 SmallRye，较新。FastAPI/Go 安全框架薄弱，需拼装。

本项目需满足等保三级，Spring Security 优势显著。

#### 维护成本

Java 人才储备最丰富、招聘成本最低，Spring 生态文档完善、社区成熟。Quarkus 学习曲线较陡。Go 业务开发人才稀缺。Python 企业级工程化经验薄弱。

### 2.4 选型结论

**选定：Spring Boot 3 + Spring Cloud Alibaba**

理由：在安全性、扩展性、维护成本三个高权重维度全面领先，企业级生态（多租户插件、Seata 分布式事务、Sentinel 熔断）与本项目需求高度契合。AI 能力通过服务化解耦（vLLM/Embedding 独立部署，Java 通过 HTTP/gRPC 调用）弥补。

## 三、前端框架选型对比

### 3.1 候选方案概览

| 方案 | 定位 | 生态特点 |
|------|------|----------|
| **Vue 3 + Element Plus** | 渐进式框架，企业后台首选 | 中文生态强，Element Plus 组件丰富 |
| React + Ant Design | 灵活度高，社区最大 | 生态最庞大，Ant Design 企业级组件 |
| Angular | 大而全，强约束 | TypeScript 原生，适合大型团队 |

### 3.2 多维度评分

| 维度 | Vue 3 | React | Angular |
|------|:---:|:---:|:---:|
| 性能 | ★★★★ | ★★★★ | ★★★★ |
| 扩展性 | ★★★★ | ★★★★★ | ★★★★ |
| 安全性 | ★★★★ | ★★★★ | ★★★★★ |
| 维护成本 | ★★★★★ | ★★★★ | ★★★ |
| **加权得分** | **4.36** | **4.20** | **3.84** |

### 3.3 详细分析

- **性能**：三者均采用虚拟 DOM + 响应式更新，Vue 3 的 Composition API + Proxy 响应式性能优秀，React Fiber 架构成熟，Angular ChangeDetection 可控。差距可忽略。
- **扩展性**：React 生态最大（Redux/Zustand/React Query），Vue 3 生态够用且中文文档完善，Angular 内置 RxJS/依赖注入，扩展方式统一但学习成本高。
- **安全性**：Angular 内置 XSS 防护（默认转义）、CSP 支持，安全性最强。Vue/React 需开发者注意 v-html/dangerouslySetInnerHTML。
- **维护成本**：Vue 3 单文件组件 + Composition API 上手快，Element Plus 中文生态对国内团队友好，招聘成本低。Angular 强约束适合大团队但学习曲线陡峭。

### 3.4 选型结论

**选定：Vue 3 + TypeScript + Vite + Element Plus**

理由：中文生态完善、Element Plus 组件覆盖企业后台 90% 场景、学习曲线平缓、国内人才储备充足。Composition API 满足复杂状态管理需求。

## 四、关系型数据库选型对比

### 4.1 候选方案概览

| 方案 | 类型 | 定位 |
|------|------|------|
| **MySQL 8.0** | 单机/主从 | 企业级关系数据库事实标准 |
| PostgreSQL 16 | 单机/扩展 | 功能最全的开源数据库 |
| OceanBase | 分布式 | 蚂蚁金融级分布式数据库 |
| TiDB | 分布式 | 兼容 MySQL 协议的分布式数据库 |

### 4.2 多维度评分

| 维度 | MySQL 8.0 | PostgreSQL | OceanBase | TiDB |
|------|:---:|:---:|:---:|:---:|
| 性能 | ★★★★ | ★★★★ | ★★★★★ | ★★★★★ |
| 扩展性 | ★★★ | ★★★ | ★★★★★ | ★★★★★ |
| 安全性 | ★★★★ | ★★★★ | ★★★★★ | ★★★★ |
| 维护成本 | ★★★★★ | ★★★★ | ★★ | ★★★ |
| **加权得分** | **4.12** | **3.72** | **3.96** | **3.96** |

### 4.3 详细分析

#### 性能与扩展性

- **MySQL**：单机性能优秀，主从架构支持读扩展，但写入扩展需分库分表（ShardingSphere），运维复杂度高。适合百万级文档量。
- **PostgreSQL**：功能最全（JSONB/数组/全文检索/向量扩展 pgvector），但高并发写入不如 MySQL，分布式扩展依赖 Citus。
- **OceanBase/TiDB**：原生分布式架构，自动分片 + 强一致性，可线性扩展至 PB 级。适合超大规模，但运维复杂度高、资源消耗大。

本项目单租户百万文档、千级并发，MySQL 主从 + 读写分离足够支撑。若未来数据量达亿级可平滑迁移 TiDB（兼容 MySQL 协议）。

#### 安全性

MySQL 企业版支持透明数据加密(TDE)、审计插件；OceanBase 金融级安全特性最全（多租户资源隔离、加密、审计一体化）。本项目等保三级需求，MySQL 企业版/社区版 + 应用层加密可满足。

#### 维护成本

MySQL 人才储备最丰富、运维工具链最成熟（MHA/Orchestrator/备份恢复），DBA 招聘容易。分布式数据库运维门槛高，需专门团队。

### 4.4 选型结论

**选定：MySQL 8.0**（主从架构 + 读写分离），预留 TiDB 平滑升级路径

理由：百万级文档量 MySQL 足够，运维成本最低，多租户行级隔离（tenant_id）方案成熟。TiDB 兼容 MySQL 协议，未来数据量增长可无缝迁移。

## 五、向量数据库选型对比

### 5.1 候选方案概览

| 方案 | 定位 | 索引类型 |
|------|------|----------|
| **Milvus 2.4** | 分布式向量数据库 | IVF/HNSW/DiskANN |
| Qdrant | Rust 高性能向量库 | HNSW |
| Weaviate | 语义搜索引擎 | HNSW |
| ChromaDB | 轻量嵌入式向量库 | HNSW |
| PgVector | PostgreSQL 向量扩展 | IVF/HNSW |

### 5.2 多维度评分

| 维度 | Milvus 2.4 | Qdrant | Weaviate | ChromaDB | PgVector |
|------|:---:|:---:|:---:|:---:|:---:|
| 性能 | ★★★★★ | ★★★★★ | ★★★★ | ★★★ | ★★★ |
| 扩展性 | ★★★★★ | ★★★ | ★★★★ | ★★ | ★★★ |
| 安全性 | ★★★★ | ★★★ | ★★★★ | ★★ | ★★★★ |
| 维护成本 | ★★★ | ★★★★ | ★★★ | ★★★★★ | ★★★★★ |
| **加权得分** | **4.32** | **3.96** | **3.72** | **3.12** | **3.84** |

### 5.3 详细分析

#### 性能

Milvus 支持十亿级向量检索，HNSW 索引毫秒级返回，DiskANN 支持超大规模低成本存储。Qdrant（Rust 实现）单机性能极强，过滤检索优秀。PgVector 依赖 PostgreSQL，万级以上性能下降明显。ChromaDB 仅适合原型验证（MVP 阶段使用 numpy 替代）。

#### 扩展性

Milvus 原生分布式架构（计算存储分离 + 分片 + 副本），可水平扩展至百节点。Qdrant 分布式版较新。PgVector 受限于 PostgreSQL 单机能力。

#### RAG 场景契合度

Milvus 支持混合检索（向量+标量过滤）、多向量字段、动态 Schema，与 RAG 的"语义检索 + 元数据过滤"需求完美契合。MVP 验证版用 numpy 自实现检索证明了链路可行，正式版升级 Milvus 可支撑万级→亿级平滑扩展。

### 5.4 选型结论

**选定：Milvus 2.4**

理由：分布式架构支撑大规模扩展，HNSW/DiskANN 索引满足性能需求，标量过滤适配多租户场景。MVP 验证的 numpy 方案接口可直接平移。

## 六、搜索引擎选型对比

### 6.1 候选方案概览

| 方案 | 定位 | 检索能力 |
|------|------|----------|
| **Elasticsearch 8** | 全文检索引擎 | 全文+向量+混合检索 |
| OpenSearch | ES 开源分支 | 同 ES |
| Meilisearch | 轻量搜索引擎 | 即时搜索 |
| Manticore | SQL 全文检索 | SQL 接口全文检索 |

### 6.2 多维度评分

| 维度 | Elasticsearch 8 | OpenSearch | Meilisearch | Manticore |
|------|:---:|:---:|:---:|:---:|
| 性能 | ★★★★★ | ★★★★★ | ★★★★ | ★★★★ |
| 扩展性 | ★★★★★ | ★★★★★ | ★★★ | ★★★ |
| 安全性 | ★★★★ | ★★★★★ | ★★★ | ★★★ |
| 维护成本 | ★★★ | ★★★ | ★★★★★ | ★★★★ |
| **加权得分** | **4.32** | **4.40** | **3.72** | **3.56** |

### 6.3 详细分析

#### 全文+向量混合检索

ES 8 内置 kNN 向量检索（dense_vector + HNSW），支持全文检索与向量检索融合（RRF），是 RAG 混合检索的理想选择。OpenSearch 为 AWS 维护的 ES 开源分支，安全特性更完善（默认启用安全插件）。

#### 资源消耗

ES 基于 JVM，内存消耗大（建议 16GB+ 堆内存），集群运维复杂。Meilisearch 轻量（Rust 实现）但功能有限，不支持复杂聚合。

#### 扩展性

ES/OpenSearch 分片+副本机制成熟，可扩展至百节点 PB 级。Meilisearch/Manticore 分布式能力弱。

### 6.4 选型结论

**选定：Elasticsearch 8**（也可考虑 OpenSearch 规避 ES 协议风险）

理由：全文检索+向量混合检索是 RAG 效果提升的关键能力，ES 生态成熟、分片扩展能力强。虽资源消耗大，但企业级部署可接受。

## 八、消息队列选型对比

### 8.1 候选方案概览

| 方案 | 定位 | 特点 |
|------|------|------|
| **RocketMQ** | 金融级消息队列 | 事务消息、顺序消息 |
| Kafka | 流处理平台 | 超高吞吐、日志场景首选 |
| RabbitMQ | 传统消息队列 | 路由灵活、AMQP 协议 |
| Pulsar | 云原生消息队列 | 计算存储分离、多租户 |

### 8.2 多维度评分

| 维度 | RocketMQ | Kafka | RabbitMQ | Pulsar |
|------|:---:|:---:|:---:|:---:|
| 性能 | ★★★★★ | ★★★★★ | ★★★ | ★★★★★ |
| 扩展性 | ★★★★ | ★★★★★ | ★★★ | ★★★★★ |
| 安全性 | ★★★★ | ★★★★ | ★★★★ | ★★★★ |
| 维护成本 | ★★★★ | ★★★ | ★★★★ | ★★★ |
| **加权得分** | **4.36** | **4.20** | **3.56** | **3.96** |

### 8.3 详细分析

文档解析→向量化→索引构建是异步流水线，需要消息队列驱动。RocketMQ 事务消息保证"文档入库+消息发送"的最终一致性，顺序消息保证同一文档的处理顺序。Kafka 吞吐更高但运维复杂（依赖 ZK/KRaft）。RabbitMQ 吞吐有限。Pulsar 架构先进但社区较小。

### 8.4 选型结论

**选定：RocketMQ**

理由：事务消息保障文档处理一致性，Spring Cloud Alibaba 生态原生集成，国内文档与人才储备充足。

MinIO 完全兼容 S3 API、部署极简（单二进制）、纠删码提供数据冗余、性能优秀。Ceph 功能最全（块/对象/文件统一）但运维极其复杂。FastDFS 停止维护。SeaweedFS 性能好但 S3 兼容不完整。

### 9.4 选型结论

**选定：MinIO**

理由：S3 兼容、部署轻量、纠删码冗余、运维简单，完美适配私有化文档存储场景。

## 十、LLM 推理引擎选型对比

### 10.1 候选方案概览

| 方案 | 定位 | 特点 |
|------|------|------|
| **vLLM** | 高吞吐推理引擎 | PagedAttention、连续批处理 |
| TGI (Text Generation Inference) | HuggingFace 推理服务 | 流式输出、张量并行 |
| Ollama | 本地模型管理 | 极简部署、OpenAI 兼容 |
| LMDeploy | 商汤推理框架 | TurboMind 引擎、量化 |
| 商业 API (DashScope) | 云端 LLM 服务 | 免部署、按量计费 |

### 10.2 多维度评分

| 维度 | vLLM | TGI | Ollama | LMDeploy | 商业API |
|------|:---:|:---:|:---:|:---:|:---:|
| 性能 | ★★★★★ | ★★★★★ | ★★★ | ★★★★★ | ★★★★★ |
| 扩展性 | ★★★★ | ★★★★ | ★★ | ★★★★ | ★★★★★ |
| 安全性 | ★★★★★ | ★★★★★ | ★★★★ | ★★★★ | ★★ |
| 维护成本 | ★★★ | ★★★ | ★★★★★ | ★★★ | ★★★★★ |
| **加权得分** | **4.36** | **4.36** | **3.72** | **4.20** | **4.12** |

### 10.3 详细分析

#### 私有化部署

vLLM 的 PagedAttention 显存效率极高（吞吐量达 HuggingFace 的 24 倍），连续批处理动态组合请求，是私有化 LLM 推理的最优解。TGI 功能相近但生态略逊。Ollama 部署极简但性能/并发能力弱，适合个人/小团队。LMDeploy 量化推理优秀但社区较小。

#### 数据安全

商业 API 需将文档发送至云端，敏感文档不合规。vLLM/TGI 私有化部署数据不出企业，满足等保要求。

#### MVP 传承

MVP 验证版通过 DashScope API + Ollama 本地验证了 RAG 链路可行性，正式版升级为 vLLM 私有化部署，商业 API 作为兜底（多供应商适配层已验证可切换）。

### 10.4 选型结论

**选定：vLLM**（私有化部署），商业 API 作为兜底

理由：PagedAttention 显存效率最优、吞吐量最高、数据不出企业满足安全合规。多供应商适配层（已在 MVP 验证）支持 vLLM 与远程 API 灵活切换。

## 十四、选型决策总结

### 14.1 核心决策原则

1. **企业级生态优先**：选择 Spring Cloud Alibaba、MySQL、Redis 等成熟企业级生态，降低长期维护风险
2. **私有化与数据安全**：所有组件支持私有化部署，数据不出企业，满足等保三级
3. **RAG 场景契合**：Milvus + ES + vLLM 组合覆盖向量检索、全文检索、LLM 推理全链路
4. **可平滑演进**：MySQL→TiDB、numpy→Milvus、API→vLLM 均预留升级路径

### 14.2 与本项目 MVP 验证版的技术传承

本项目当前为 MVP 验证版（Python + FastAPI + numpy），以下技术结论可直接沉淀至未来企业版：

| 维度 | 本项目（MVP） | 企业版演进方向 | 传承关系 |
|------|-----------|--------|----------|
| 语言 | Python | Java（主）+ Python（AI 边车） | MVP 的 AI 调用接口作为边车原型 |
| 向量存储 | numpy + JSON | Milvus | 检索接口兼容，数据可迁移 |
| LLM | 多供应商适配层 | vLLM + API 兜底 | 适配层直接复用 |
| 分块策略 | 递归分块器 | 同 | 验证最优参数直接沉淀 |
| 提示词 | System Prompt | 同 | 防幻觉范式固化 |

本项目验证的技术结论（分块策略、检索效果、提示词模板）可直接沉淀到企业版设计，技术选型决策有据可依。

### 14.3 演进建议

1. **分阶段引入**：先 MySQL + Redis + Milvus + MinIO 核心组件，ES/RocketMQ 按需引入
2. **持续评估**：每半年复盘技术栈，关注 Dragonfly/Pulsar/OpenSearch 等新兴方案成熟度
3. **保持可替换**：通过适配层（已在 MVP 验证）保持 LLM/Embedding 供应商可切换，避免厂商锁定
